# CLAUDE.md 模板：牛马库

> 用法：把下面内容复制到项目根目录 `CLAUDE.md`，再按你们团队的实际分工微调。

```md
# 牛马库协作说明

## 你正在维护什么
你正在维护 KA21 AI 牛马库主站。它不是“AI 工具堆砌页”，而是一个强调真实体验、真实筛选、真实推荐的内容型产品。

当前仓库核心技术与形态：
- Next.js App Router
- React + TypeScript
- next-intl 多语言
- Tailwind CSS
- Jest 测试
- Cloudflare / OpenNext 部署链路
- Web + 小程序共享部分数据链路

你的首要目标不是堆功能，而是：
1. 帮用户更快找到真正好用的 AI 工具、教程和活动内容
2. 保护站点的数据正确性、结构稳定性和长期可维护性
3. 在不破坏现有体验的前提下持续优化

## 产品原则
- 用户信任高于一切，宁可少推荐，也不要乱推荐
- 数据正确比页面花哨更重要
- 小步改动、快速验证，避免大范围无把握重写
- 优先复用现有模式，避免额外引入平行体系
- 移动端可用性、搜索效率、信息清晰度优先

## 这个仓库里的关键 Source Of Truth
- `src/data/tools.json`：工具数据主源
- `src/data/tutorials.json`：教程数据主源
- `src/data/featured.json`：首页精选与热门配置
- `src/data/devLogs.ts` 与 `src/data/devlog-submissions.json`：开发日志内容
- `messages/zh.json` 与 `messages/en.json`：界面文案国际化
- `src/app/[locale]/`：主站页面入口
- `src/lib/`：数据映射、校验、安全、配置逻辑
- `miniprogram/`：小程序端消费的数据与页面

如果任务涉及新增、删除、排序、推荐、筛选、跳转、详情页展示，先找到对应的 Source Of Truth，不要只改展示层。

## 你做改动时必须遵守的约束

### 1. 数据约束
- 工具、教程、活动、开发日志的 `id` 一旦发布，默认视为稳定标识，不要轻易改名
- 改 `featured` 前，先确认对应 ID 真实存在
- 教程与工具的关联关系要双向自洽，不要出现展示层有卡片但数据层找不到实体
- 不要凭空补造工具能力、价格、平台可用性等信息
- 如果是整理内容，优先修正结构与表达，不要随意篡改原始事实

### 2. 国际化约束
- 所有用户可见文案都要考虑中英文一致性
- 不要只改中文页面忘了英文
- 不要在多语言页面里硬编码文案，优先走 `messages/*.json` 或双语数据字段
- 涉及路由和链接时，优先沿用现有 locale 方案，不要私自发明新的语言切换逻辑

### 3. 体验约束
- 首页、搜索、分类、详情页都属于高流量核心路径，改动要克制
- 默认保持现有视觉语言，除非任务明确要求重设计
- 优先保证信息层级清楚、卡片可扫读、筛选可理解、手机可用
- 不要为了“更智能”牺牲可预测性

### 4. 工程约束
- 优先数据驱动，不要把静态数据写死在组件里
- 优先局部改动，不要无必要跨很多文件同步重构
- 尽量沿用现有组件、工具函数和类型定义
- 涉及安全、管理接口、写回 GitHub 的逻辑时，默认保守处理

## 常见任务处理原则

### 新增工具或教程
- 先改数据源，再补展示层
- 补齐必要字段，不要只填最少字段让后续页面兜底
- 检查图标、标签、分类、相关推荐是否完整
- 如果会影响小程序数据，记得同步导出链路

### 调整首页或热门区
- 优先检查 `src/data/featured.json`
- 不要只改卡片样式却忽略数据顺序和兜底逻辑
- 任何“换一换”“推荐池”相关调整，都要先考虑数据是否足够支撑

### 修多语言问题
- 先定位是 messages 缺键、路由问题、还是数据层单双语不一致
- 不要用“临时硬编码英文”当最终解法

### 修搜索/筛选问题
- 先看数据映射和查询逻辑
- 不要只在 UI 层打补丁

## 完成任务后的默认检查

### 数据或配置改动
- `npm run validate-featured`
- `npm run validate-tutorial-relations`
- `npm run check-i18n-keys`

### 组件、交互、页面逻辑改动
- `npm test`
- `npm run lint`

### 较大改动或准备上线
- `npm run build`

### 涉及 Cloudflare / OpenNext 链路
- `npm run build:cloudflare`

### 涉及小程序共享数据
- 评估是否需要 `npm run export:miniapp-data`
- 如依赖教程全文缓存，再评估是否需要 `npm run cache:miniapp-fulltext`

## 明确不要做的事
- 不要为了省事删除现有数据字段或批量改 ID
- 不要只修表面 UI，不修底层数据问题
- 不要无请求地大改首页视觉风格
- 不要把临时脚本结果当作长期数据源
- 不要跳过中英文一致性检查
- 不要在没有验证的情况下改动会影响小程序的数据格式

## 希望你输出结果时做到
- 先说你改了什么，再说为什么这么改
- 如果有假设，明确写出来
- 如果没跑某个验证命令，明确说明原因
- 如果发现任务背后是数据问题、结构问题或规范问题，要直接指出，不要只做表面修补

## 这份站点最重要的成功标准
- 用户能更快找到值得用的工具
- 数据是可信的
- 结构是稳定的
- 中英文是一致的
- 页面在手机上也顺手
```
