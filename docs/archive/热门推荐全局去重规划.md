# 热门推荐板块全局去重实现规划

版本：v1.0  
日期：2025-01-27  
状态：📋 规划阶段（补完版）

## 1. 问题描述

### 当前问题
热门推荐板块内部三个分类之间存在工具重复显示的问题：
- 用户在第一个分类点击"换一换"时，可能会看到第二个或第三个分类中已经显示的工具
- 当前设计：**分类内去重；跨分类允许重复**
- 影响用户体验，造成困惑

### 目标
实现热门推荐板块内部的全局去重，确保三个分类之间不会重复显示相同的工具。

### 范围澄清（重要）
- 仅在“热门推荐”板块的三个分类（列）之间去重；不与页面下方其他模块进行去重。
- 首屏 SSR 行为：
  - 方案1（推荐）：SSR 允许临时重复，水合后（客户端）进行全局去重并稳定显示。
  - 方案2（严格）：SSR 即进行去重排布，避免首屏重复，但需在服务端具备同等的全局状态计算（工作量更大）。
  - 当前阶段采用方案1，后续可视业务需要升级至方案2。

## 2. 技术分析

### 现有架构
```
FeaturedToolsManager (src/lib/featured.ts)
├── 分类内去重：sessionStorage存储每个分类的已显示工具
├── 洗牌算法：Fisher-Yates算法
├── 分页管理：支持换一换功能
└── 兜底机制：backup_pool补充工具

HotSection (src/components/hot/HotSection.tsx)
├── 管理三个分类的状态
├── 初始化FeaturedToolsManager
└── 传递数据给HotCategory

HotCategory (src/components/hot/HotCategory.tsx)
├── 独立的换一换逻辑
├── 调用FeaturedToolsManager.getPaginatedTools
└── 更新本地状态
```

### 核心问题
- 每个分类独立管理自己的状态
- 没有跨分类的全局状态同步
- `getPaginatedTools`方法只考虑分类内去重

## 3. 实现方案

### 方案设计
**全局去重 + 主分类锁定**：在热门推荐板块内部实现跨分类去重，并为每个工具确定唯一“主分类”（Category Locking），杜绝跨分类“漂移”。

### 技术实现
1. **全局状态管理（单例）**：在 `HotSection` 中创建单例 `FeaturedToolsManager`，通过 props 下发给每个 `HotCategory`，集中管理全局状态与分页状态。
2. **主分类锁定（核心）**：为每个工具确定唯一主分类（会话内固定），规则为：featured > 匹配度（tags/categories）> `config.order`。优先使用 `tool.hotPrimaryCategory`（如存在），否则运行时计算并写入 `sessionStorage`。
3. **选片来源变更**：每个分类仅从“归属于该分类”的工具池中选取（先 featured，再备选）；不再跨分类借用 `allTools` 兜底。
4. **去重逻辑（安全网）**：全局当前展示仍用于排除，但理论上由“主分类锁定”自然满足。
5. **不足策略**：若某分类可用工具不足以填满 `pageSize`，允许少于 `pageSize` 展示，不跨分类兜底。
6. **替代方案（不选）**：保留各分类独立管理器实例，通过共享 `sessionStorage` 的全局键读写同步。权衡后不选，避免多实例竞争与一致性问题。

## 5. 技术细节

### 数据结构设计（全局状态）
```typescript
// 持久化键名约定
// - 分页状态：  hotrec:${categoryKey}:state
// - 全局当前：  hotrec:global:current
// - 主分类映射：hotrec:global:assignment

interface GlobalDisplayState {
  // 各分类当前显示的工具（当前页）
  categoryCurrentTools: Record<string, string[]>; // { [categoryKey]: toolIds[] }
}

// 工具结构扩展（长期固定可选）
interface Tool {
  // ...
  hotPrimaryCategory?: string; // 若存在，优先作为其主分类（支持 SSR）
}
```

### 核心方法设计（共享管理器 + 主分类锁定）

### 状态同步机制
1. **初始化时**：HotSection 创建管理器 → 构建/加载主分类映射（优先 `tool.hotPrimaryCategory`，缺失则计算并缓存至 `hotrec:global:assignment`）→ 拉取各分类首屏 → 写入 `hotrec:global:current`。
2. **水合后**：各分类从共享管理器拿到首屏（若 SSR 有重复，客户端进行校正）。
3. **换一换时**：仅从本分类归属池取下一页候选（排除其它分类当前页）→ 更新分页状态 → 回写 `hotrec:global:current`。
4. **不广播刷新**：不强制刷新其它分类，避免抖动；其它分类在自身刷新时自然读取最新全局状态。

### 选页与兜底算法（细则）
1. 构建候选：按“2 精选 + 1 非精选”的策略生成当前页候选（首页固定，其它页取 deck）。
2. 计算排除集：聚合“其它分类当前展示”的工具 ID 集合（Set）。
3. 过滤候选：去除与排除集重叠的工具。
4. 数量不足时仅在本分类归属池内补齐：
   - 用当前 deck 的剩余顺序补齐；
   - 仍不足 → 用本分类 `backupPool`（归属本分类的工具）补齐；
   - 仍不足 → 允许少于 `pageSize`（不跨分类借用，不用 `allTools` 兜底）。
5. featured 重叠处理：若某工具在多个分类均为 featured，只允许当前一个分类露出；其他分类以“下一精选或非精选”替换。
6. 写回：成功产出新页后，调用 `updateGlobalDisplayState(categoryKey, newPageIds)`。

### SSR 行为
- 若工具包含 `hotPrimaryCategory`，SSR 即可稳定无漂移；
- 否则默认采用“SSR 可暂时重复，客户端水合后进行主分类锁定与去重”的策略，避免服务端引入复杂的全局状态计算。

## 6. 风险评估

### 技术风险
- **状态同步复杂性**：需要确保三个分类的状态实时同步
- **性能影响**：全局状态管理可能增加计算开销
- **兼容性问题**：需要保持现有API的向后兼容
 - **多实例竞争**：如果误用多实例管理器，可能产生写入覆盖或状态抖动
 - **分类工具不足**：在不跨分类兜底的策略下，可能出现未满页的情况

### 缓解措施
- 使用React的useState和useEffect进行状态管理
- 优化算法复杂度，避免不必要的计算
 - 保持现有接口不变，内部实现优化
 - 采用单例管理器（HotSection 提供），不广播刷新，按需读取最新全局状态
 - 产品确定允许少于 `pageSize` 展示；中长期通过数据侧补全各分类工具量
