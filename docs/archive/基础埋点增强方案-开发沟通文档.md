## 📋 方案概述

### 什么是基础埋点增强方案？

简单来说，就是**升级我们的数据收集系统**，从"知道用户做了什么"升级到"深度了解用户行为"。

**打个比方**：
- **之前**：像简单的收银台，只记录"买了什么、什么时候买的"
- **现在**：像智能商场，记录"顾客怎么逛的、在哪停留、为什么选择、完整路径"

### 核心目标

1. **数据更全面**：收集更多维度的用户行为数据
2. **分析更精准**：为热度排序和个性化推荐提供数据基础
3. **决策更科学**：用数据驱动产品优化，而不是靠猜测

### 3. 增加曝光埋点

**目标**：知道用户"看到了但没点击"的工具

**业务价值**：
- 计算真实的点击率（CTR = 点击数 / 曝光数）
- 发现有潜力但位置不好的工具

**技术实现**：
```javascript
// 使用 Intersection Observer API
class ExposureTracker {
  constructor() {
    this.observer = new IntersectionObserver(
      this.handleIntersection.bind(this),
      {
        threshold: 0.5, // 50%可见时触发
        rootMargin: '0px'
      }
    );
  }
  
  handleIntersection(entries) {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        // 开始曝光计时
        this.startExposureTimer(entry.target);
      } else {
        // 停止曝光计时
        this.stopExposureTimer(entry.target);
      }
    });
  }
  
  recordExposure(toolId, duration) {
    trackEvent('tool_exposure', {
      tool_id: toolId,
      exposure_duration: duration, // 曝光时长
      visibility_percentage: 50,   // 可见百分比
      viewport_position: getElementPosition(),
      timestamp: Date.now()
    });
  }
}
```

**开发工作量**：2-3天

### 4. 用户行为序列追踪

**目标**：记录用户的完整行为路径

**业务价值**：了解用户决策过程，优化页面布局

**实现方案**：
```javascript
class BehaviorTracker {
  constructor() {
    this.behaviorSequence = [];
    this.sessionId = generateSessionId();
  }
  
  recordBehavior(action, data) {
    this.behaviorSequence.push({
      action: action,           // 'scroll', 'hover', 'click', 'page_enter'
      data: data,
      timestamp: Date.now(),
      sequence_index: this.behaviorSequence.length
    });
    
    // 达到一定数量后批量上报
    if (this.behaviorSequence.length >= 10) {
      this.flushBehaviors();
    }
  }
  
  // 监听各种用户行为
  bindEvents() {
    // 滚动行为
    window.addEventListener('scroll', () => {
      this.recordBehavior('scroll', {
        position: window.pageYOffset,
        direction: this.getScrollDirection()
      });
    });
    
    // 鼠标悬停
    document.addEventListener('mouseover', (e) => {
      if (e.target.closest('.tool-item')) {
        this.recordBehavior('hover', {
          tool_id: getToolId(e.target),
          hover_start: Date.now()
        });
      }
    });
  }
}
```

**开发工作量**：3-4天

### 5. 数据质量检查机制

**目标**：确保收集的数据是准确可靠的

**业务价值**：避免基于错误数据做决策

**实现方案**：
```javascript
class DataValidator {
  validateEvent(eventType, eventData) {
    const rules = this.getValidationRules(eventType);
    const errors = [];
    
    // 检查必填字段
    rules.required.forEach(field => {
      if (!eventData[field]) {
        errors.push(`Missing required field: ${field}`);
      }
    });
    
    // 检查数据类型
    Object.entries(rules.types).forEach(([field, type]) => {
      if (eventData[field] && typeof eventData[field] !== type) {
        errors.push(`Invalid type for ${field}: expected ${type}`);
      }
    });
    
    // 检查数值范围
    Object.entries(rules.ranges).forEach(([field, range]) => {
      const value = eventData[field];
      if (value < range.min || value > range.max) {
        errors.push(`Value out of range for ${field}`);
      }
    });
    
    return errors.length === 0;
  }
  
  getValidationRules(eventType) {
    return {
      tool_click_enhanced: {
        required: ['tool_id', 'category', 'position', 'timestamp'],
        types: {
          tool_id: 'string',
          position: 'number',
          timestamp: 'number'
        },
        ranges: {
          position: { min: 1, max: 20 },
          timestamp: { min: Date.now() - 86400000, max: Date.now() + 3600000 }
        }
      }
    };
  }
}

// 集成到埋点流程
function trackEvent(eventType, eventData) {
  if (dataValidator.validateEvent(eventType, eventData)) {
    // 数据有效，发送埋点
    sendToAnalytics(eventType, eventData);
  } else {
    // 数据无效，记录错误但不发送
    console.warn('Invalid event data:', eventType, eventData);
  }
}
```

**开发工作量**：2-3天

  constructor() {
    this.queue = [];
    this.batchSize = 10;
    this.flushInterval = 30000; // 30秒
  }
  
  add(event) {
    this.queue.push(event);
    
    if (this.queue.length >= this.batchSize) {
      this.flush();
    }
  }
  
  flush() {
    if (this.queue.length > 0) {
      sendBatchEvents(this.queue);
      this.queue = [];

  }

## ⚠️ 风险和注意事项

### 1. 性能风险

**风险**：埋点过多可能影响页面性能

**缓解措施**：
- 使用事件节流和防抖
- 批量上报，减少网络请求
- 异步处理，不阻塞主线程
- 监控页面性能指标

### 2. 数据隐私

**风险**：收集过多用户数据可能涉及隐私问题

**缓解措施**：
- 不收集敏感个人信息
- 用户ID使用加密或匿名化
- 遵循数据保护法规
- 提供数据删除机制

### 3. 开发复杂度

**风险**：增加代码复杂度，可能引入bug

**缓解措施**：
- 模块化设计，职责清晰
- 完整的单元测试
- 渐进式实施，分步验证
- 详细的错误日志

## 🧭 Umami 适配版：背景、现状、价值与 TODO 计划（不写代码）

本节为当前项目的落地补充，结合我们的实际技术栈与统计平台（Umami Cloud），明确为什么要加、加什么、怎么验收。仅文档对齐，不涉及代码改动。

### 背景
- 无自建后端/数据库，统计依赖 Umami Cloud。
- 业务诉求：评估首页“热门推荐”区的有效性（位次/分类/标签对点击的影响），支撑排序优化与文案迭代。
- 约束：Umami 的事件/属性以轻量为宜，原生对跨事件计算（如 CTR）支持有限；路径/回放由 Clarity 承担更合适。

### 现状（代码与数据核查）
- 已接入：全站 Umami 脚本（`src/components/UmamiAnalytics.tsx`）。
- 热门区：通过 `src/lib/hot-analytics.ts` 上报 `hot_click` 与 `hot_shuffle` 到 Umami；字段含 `category`、`tool_id`、`position`、`page_index`、`session_id`。
- 非热门区：`ToolCard` 点击仅在 Clarity 统计，未统一到 Umami，因此 Umami 中点击总体偏低。
- 近 7 天（示例）：`hot_shuffle=133`、`hot_click=8`；属性统计显示 `category/position/tool_id/page_index/session_id` 均有值。
- 缺口：
  - 无稳定的 `category_key`（仅中文 `category`），不利于长期对比与查询稳定性。
  - 无 `from` 来源字段（虽在代码常量存在，但未上报到 Umami）。
  - 未覆盖 `tag_click`；未统一非热门区点击到 Umami；未做曝光（CTR）。

### 为什么要添加（价值）
- 排序与选品优化：基于 `position/category/tool_id` 的点击分布，直接在 Umami 里对比，指导卡片顺序与选品。
- 标签策略验证：新增 `tag_click`，量化“热门/推荐”等标签的真实贡献，优化文案与样式。
- 口径统一与对比：将非热门区点击统一到 Umami，形成整站一致视图，便于横向比较与归因。
- 成本可控：不做大规模曝光/行为序列，专注 Umami 能直接产出洞察的少量高价值事件。

### 范围与原则（Umami-first，精简高 ROI）
- 事件保持轻量：每事件 5–8 个关键属性，数值/短字符串为主。
- 命名统一：`category_key`（稳定英文 key）+ `category`（中文展示），`position` 1 基。
- 覆盖最小闭环：`hot_click`、`hot_shuffle`、新增 `tag_click`、可选 `view_all_click`。
- 曝光/CTR：默认暂缓；如确有需求，采用“强去重 + 阈值 + 采样”的最小化方案（见下）。

### 事件 Schema 草案（文档对齐，非实现）
- hot_click
  - 必填：`category_key`、`category`、`tool_id`、`position`
  - 可选：`from='hot_section'`
  - 口径：点击热门推荐卡片时触发；同一次用户动作仅一次；`position` 从 1 开始。
- hot_shuffle
  - 必填：`category_key`、`category`、`page_index`
  - 可选：`page_size`（如果需要还原上下文）
  - 口径：点击“换一换”后触发；`page_index` 为换完后的页。
- tag_click（新增）
  - 必填：`tool_id`、`tag_type`（如 hot/new/recommend）、`tag_text`、`position`、`category_key`、`category`
  - 口径：点击工具卡片上的标签时触发；需避免与卡片点击重复上报（事件冒泡处理）。
- view_all_click（可选）
  - 必填：`category_key`、`category`
  - 口径：点击“查看全部/更多”等导流入口时触发。

> 说明：不额外引入自定义 `session_id` 作为分析基准，Umami 自带会话体系即可；若作为辅助维度保留，不作为必填。

### 曝光/CTR（如必须要）
- 默认不做；若确有业务刚需，采用最小口径：
  - 触发条件：≥50% 可见且连续 ≥1s；同一 `page_view` 仅一次有效曝光；列表容器使用 IntersectionObserver。
  - 属性：`category_key`、`category`、`tool_id`、`position`（可选 `list_id`）。
  - 控制：强去重 + 采样（如 10–30%）+ 上限（每页最多 N 条），避免事件爆量与成本上升。
