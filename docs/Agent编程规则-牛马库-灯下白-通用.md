# Agent 编程规则（牛马库 / 灯下白 / 通用）

> 这份文档基于《盘一下Claude Code源码里的8个隐藏新功能》整理而来，但不把文中的未上线或泄露功能当成事实承诺，而是把它们转译成一套更稳、更能长期复用的编程规则。
>
> 适用范围：
> - 牛马库主站
> - 《灯下白》播客数据链路
> - 其他需要用 Codex / Claude 辅助编程的项目

## 这份规则的核心目标

- 让 AI 不只是“会写代码”，而是能参与规划、执行、复盘、交接和长期维护
- 让复杂任务变成可拆解、可恢复、可验证的工作流
- 让项目知识不只存在聊天记录里，而能沉淀为可复用规则
- 让主动性建立在信号和边界上，而不是建立在自嗨和过度自动化上

## 总原则

- 把文章里的“炫功能”翻译成“硬规则”
- 先追求稳定工作流，再追求花哨能力
- 任何自动化都必须可解释、可暂停、可恢复
- 任何跨会话协作都必须可交接，不能靠记忆硬撑
- 任何关键改动都要经过多视角审查，而不是只看代码能不能跑

## 八条核心规则

## 1. 先做深规划，再做快执行

- 遇到高风险、高耦合、跨多文件、跨数据流的任务，先做一轮深入规划，再开始改代码
- 规划至少要回答 4 件事：目标是什么、Source of Truth 在哪、最大风险是什么、怎么验证成功
- 不允许一上来就在 UI 层打补丁，尤其是牛马库的内容流和灯下白的数据链路

牛马库重点：
- 先确认数据主源是 `src/data/*`、`messages/*`、还是 `src/lib/*`
- 先确认是否会影响多语言、首页流量路径、小程序共享数据

灯下白重点：
- 先定位问题在抓取、落盘、API、展示、刷新安全中的哪一层
- 不允许把链路问题误判成单纯前端问题

## 2. 长任务要后台化，并且支持恢复

- 只要任务可能超过一次会话，就要设计成可中断、可恢复、可重跑
- 脚本优先支持 `dry-run`、日志输出、明确错误、幂等执行
- 不要把关键进度只留在临时终端状态里

适用场景：
- 灯下白快照抓取
- 批量导出
- 构建与部署链路
- 数据清洗
- 长时间 review 或验证

落地要求：
- 输出文件路径明确
- 中间状态可检查
- 失败后能从最近一步继续，而不是全部重来

## 3. 记忆要沉淀，也要定期整理

- 会话里的结论、假设、坑点、边界条件，不能只留在聊天记录里
- 每次做完复杂任务，都要至少沉淀以下内容：做了什么、为什么这么做、哪些地方仍不稳定
- 记忆不是越多越好，关键是可检索、可删减、不过期

推荐沉淀内容：
- 决策原因
- 已验证命令
- 失败案例
- 已知限制
- 后续接手人最需要知道的上下文

## 4. 跨会话协作必须可交接

- 默认假设下一个接手的 Agent 或人不知道你刚刚做过什么
- 每个复杂任务都要留下可交接信息，而不是让下一次会话重新猜
- 交接信息至少要包含：当前状态、下一步、风险点、验证方式

好交接的标准：
- 别人不看完整聊天，也知道从哪继续
- 别人不依赖你的“脑内上下文”，也能继续推进
- 关键命令、关键文件、关键数据点都能快速找到

## 5. 工作状态必须可迁移

- 环境、命令、上下文、规则、输出路径都应该尽量做到“换机器、换会话也能继续”
- 不要把项目推进依赖在某台机器的私有状态、某个浏览器 tab、某段没有落盘的思路上
- 所有重要流程都应尽量文档化、脚本化、显式化

落实方式：
- 尽量使用仓库内脚本和文档，而不是口头约定
- 环境变量、依赖、构建方式、验证方式都写清楚
- 输出物要可追踪，不做“我这边本地已经好了”的黑箱状态

## 6. 主动性来自信号，不来自乱动

- Agent 应该主动，但主动的前提是有明确信号和边界
- 没有信号就不要乱改；有信号时也要先判断是否值得行动
- 主动模式适合提醒、扫描、预警、补充检查，不适合擅自重构

推荐主动信号：
- 校验脚本失败
- 新增数据但未同步多语言
- 灯下白最新快照日期落后
- 刷新接口异常
- 构建通过但关键页面数据为空

不推荐的主动行为：
- 未经确认就大改首页视觉
- 没有根因判断就批量改数据
- 只因为“可能更智能”就引入新系统

## 7. 关键改动必须多视角审查

- 复杂任务不能只做单线程自检，至少要做多角度 review
- 审查视角应覆盖产品、数据、工程、体验、安全，而不是只有“能运行”
- 大改动应默认先做规划审查，再做实现审查，最后做发布前审查

牛马库推荐 review 视角：
- 数据是否可信
- 中英文是否一致
- Source of Truth 是否被绕开
- 移动端和核心路径是否受影响
- 小程序共享数据是否被波及

灯下白推荐 review 视角：
- 公开数据是否真实
- 历史快照是否兼容
- 刷新接口是否更危险了
- 问题到底在抓取、落盘、API 还是 UI
- 页面看到的日期和数据是否真是最新

## 8. 给长期编程保留反馈感和动能

- 长期项目不能只靠意志力推进，要让进展“可见”
- 把小胜利显式记录下来，会比机械堆任务更能维持节奏
- 规则不是只防错，也要帮助自己持续做下去

适合显式反馈的内容：
- 本周闭环了哪条链路
- 哪个坑终于找到根因
- 哪个脚本终于稳定
- 哪次 review 挡住了风险
- 哪条规则让返工变少了

这条规则对应的不是“宠物感”，而是“长期开发动能管理”。

## Codex 与 Claude 的默认分工

Claude 更适合：
- 做需求收敛
- 做竞品理解和方案比较
- 写规划、规则、PRD、复盘
- 在复杂问题上先给结构化判断

Codex 更适合：
- 读代码和定位真实链路
- 改代码、补脚本、跑验证
- 把规则落成实际文件、命令、流程
- 做带上下文的工程执行

推荐协作顺序：
- 先让 Claude 做结构化规划或优先级判断
- 再让 Codex 进入仓库执行、验证和落盘
- 关键任务结束后，再让 Claude 帮你做复盘和规则沉淀

## 默认工作流

### 第一步：任务定义

- 目标是什么
- 为什么现在做
- 结果以什么为准

### 第二步：定位 Source of Truth

- 数据在哪
- 逻辑在哪
- 展示在哪
- 权限和部署边界在哪

### 第三步：做一轮深规划

- 拆子任务
- 标风险
- 定验证
- 判断是否需要先做 dry-run

### 第四步：执行时保持可恢复

- 小步改动
- 明确日志
- 不中断上下文
- 关键结果落盘

### 第五步：做多视角 review

- 产品是否更清楚
- 数据是否更可信
- 结构是否更稳定
- 安全是否更保守
- 体验是否更顺手

### 第六步：做交接与沉淀

- 写清楚改了什么
- 写清楚没改什么
- 写清楚剩余风险
- 把可复用经验升级成规则

## 牛马库专项规则补充

- 这是内容型产品，不是单纯 UI 项目
- 用户信任高于页面花哨
- 工具、教程、开发日志等内容默认先看数据主源，不先改展示层
- 中英文一致性和移动端可读性是默认项，不是加分项
- 任何首页、搜索、推荐相关改动，都要先检查排序逻辑和数据支持能力

## 灯下白专项规则补充

- 这是“抓取 -> 快照 -> API -> 展示 -> 刷新”的闭环，不是普通页面
- 拿不到的数据就明确拿不到，不伪造、不补零、不猜测
- 历史快照优先追加，不随意覆盖
- 刷新能力默认保守，不为方便牺牲安全边界
- 每次动抓取和刷新链路，都应明确说明验证范围和剩余不稳定点

## 明确不要做的事

- 不要把文章中的未上线能力当成真实可依赖功能
- 不要因为看见“多 Agent”就默认一切都该并行化
- 不要为了炫技引入超出团队维护能力的自动化
- 不要只追求“更主动”，却不定义触发条件和停止条件
- 不要让项目知识只存在于对话，而不进入规则、文档、脚本或数据结构

## 一句话版本

- 先规划
- 可恢复
- 能交接
- 可迁移
- 有边界地主动
- 多视角审查
- 持续沉淀
- 让长期开发有反馈感
