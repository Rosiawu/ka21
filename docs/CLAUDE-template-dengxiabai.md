```md
# 灯下白协作说明

## 你正在维护什么
你正在维护《灯下白》播客的数据追踪与展示链路。它不是普通展示页，而是一条“公开数据抓取 -> 快照落盘 -> API 提供 -> 看板展示 -> 手动刷新”的闭环。

当前关键模块包括：
- `data/podcast-dashboard/config.json`
- `data/podcast-dashboard/episodes-cache.json`
- `data/podcast-dashboard/snapshots.json`
- `scripts/podcast/capture-snapshot.ts`
- `src/app/api/podcast/dashboard/route.ts`
- `public/podcast-dashboard/index.html`
- `public/podcast-dashboard/app.js`
- `public/podcast-dashboard/styles.css`
- `src/app/[locale]/podcast/page.tsx`

你的目标不是把页面做得花，而是：
1. 让公开可见的播客数据稳定抓到
2. 让快照结构长期可维护
3. 让网页能可靠展示最新数据
4. 让“立即刷新”这条链路既可用又安全

## 产品原则
- 真实性高于好看，拿不到的数据就明确说拿不到
- 快照可靠高于临时炫技
- 不伪造增长，不用猜测补齐缺失值
- 先保闭环，再做分析增强
- 手机可看、可刷、可理解，和桌面端同样重要

## Source Of Truth
- 平台列表与平台 ID：`data/podcast-dashboard/config.json`
- 历史快照：`data/podcast-dashboard/snapshots.json`
- RSS 导出的节目基础信息缓存：`data/podcast-dashboard/episodes-cache.json`
- 抓取逻辑：`scripts/podcast/capture-snapshot.ts`
- 外部读取与刷新接口：`src/app/api/podcast/dashboard/route.ts`
- 看板展示层：`public/podcast-dashboard/*`

任何涉及平台、播放量、日期、刷新、快照结构的任务，先确认改动发生在哪个 Source Of Truth，不要只改页面。

## 你做改动时必须遵守的约束

### 1. 数据真实性
- 只记录公开可获取的数据
- 如果某个平台当前拿不到公开播放量，明确标记为不支持或抓取失败，不要默默写成 0
- 不要把估算值、私有后台数据、人工猜测混进公开快照
- 历史快照默认只追加，不随意重写；除非是在修复已确认的数据错误

### 2. 平台稳定性
- `config.json` 里的平台 `id` 默认视为稳定主键，不要轻易改
- 新增平台时，要同时考虑抓取逻辑、数据结构、前端展示、历史兼容
- 下线平台时，不要让旧快照直接失效

### 3. 抓取鲁棒性
- 网络抖动、平台超时、页面结构变化都属于常态，不是例外
- 抓取失败时要可解释，错误信息要帮助定位问题
- 优先做“可恢复的失败”，不要静默成功
- 如果抓取逻辑回退到 curl 或其他兜底方式，保持行为一致

### 4. 刷新链路安全
- 不要随意放宽 `POST /api/podcast/dashboard` 的访问范围
- 保留同源受信刷新和管理员访问的边界
- 任何刷新能力增强，都要先评估是否会把内部接口暴露给任意外部调用
- 如果任务涉及鉴权或限流，默认保守处理

### 5. 展示层约束
- 看板首先是“读数工具”，其次才是“品牌页面”
- 总播放、平台分布、日期和刷新状态应优先清晰可见
- 不要为了视觉重构牺牲数据可读性
- 改静态看板时同时考虑桌面端和移动端

## 常见任务处理原则

### 新增或修复平台抓取
- 先确认平台公开页是否真的有公开可抓数据
- 再补脚本逻辑，不要先写 UI
- 输出结果时说明平台是“支持抓取”“暂不支持”还是“当前失效”

### 修快照异常
- 先判断是抓取异常、数据写入异常、API 返回异常还是前端展示异常
- 不要一上来手改 `snapshots.json` 掩盖根因

### 修“立即刷新”按钮
- 同时检查前端请求头、同源校验、管理员保护、限流与并发锁
- 不要只看按钮点击有没有触发

### 增强数据分析展示
- 先保证原始数据对，再加趋势、洞察、解释层
- 分析文案要基于已有快照，不要演绎过度

## 完成任务后的默认检查

### 抓取或数据结构改动
- `npm run podcast:capture-snapshot -- --dry-run`
- 必要时再运行 `npm run podcast:capture-snapshot`
- 检查 `data/podcast-dashboard/*.json` 是否符合预期

### API 或页面逻辑改动
- `npm test`
- `npm run lint`

### 涉及路由、运行时或部署链路
- `npm run build`

## 明确不要做的事
- 不要把抓不到的数据伪装成 0
- 不要把历史快照当缓存随便覆盖
- 不要破坏平台 ID 稳定性
- 不要把管理员接口不小心改成公网可刷
- 不要只修前端显示，不追根因
- 不要在没有验证的情况下上线抓取规则改动

## 希望你输出结果时做到
- 说明问题发生在抓取、落盘、接口还是展示层
- 说明这次改动有没有影响历史快照兼容性
- 如果抓取仍有不稳定点，明确列出来
- 如果没跑真实抓取，只做了 dry-run，要明确说明

## 这条链路最重要的成功标准
- 最新快照能稳定生成
- 数据和日期可信
- 看板能稳定读到最新快照
- “立即刷新”既可用又不过度开放
- 用户能用它看趋势，而不是只看到一堆数字
```
